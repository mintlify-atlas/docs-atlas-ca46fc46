## Initial setup

<Steps>
  <Step title="Clone repository">
    ```bash
    git clone https://github.com/gitmaan/stellaris-companion.git
    cd stellaris-companion
    ```
  </Step>
  
  <Step title="Create Python virtual environment">
    ```bash
    python3 -m venv venv
    source venv/bin/activate  # On Windows: venv\Scripts\activate
    ```
  </Step>
  
  <Step title="Install Python dependencies">
    ```bash
    pip install .
    ```
    
    For development dependencies:
    ```bash
    pip install -e ".[dev]"
    ```
  </Step>
  
  <Step title="Build Rust parser">
    ```bash
    cd stellaris-parser
    cargo build --release
    cd ..
    ```
  </Step>
  
  <Step title="Install Node dependencies">
    ```bash
    npm -C electron install
    npm -C electron/renderer install
    ```
  </Step>
  
  <Step title="Configure environment">
    ```bash
    cp .env.example .env
    # Edit .env and add your GOOGLE_API_KEY
    ```
  </Step>
</Steps>

## Running the application

### Quick start (recommended)

The `dev.sh` script provides a one-command development flow:

```bash
./dev.sh
```

This script:
- Loads environment variables from `.env`
- Generates `STELLARIS_API_TOKEN` if missing
- Starts the Python backend on `127.0.0.1:8742`
- Waits for backend health check
- Launches Electron with renderer hot reload

<Note>
The dev script uses `lsof` to clean up orphaned backend processes from previous runs.
</Note>

### Manual startup

For debugging or separate terminal workflows:

<Tabs>
  <Tab title="Backend">
    ```bash
    python3 -m backend.electron_main --host 127.0.0.1 --port 8742
    ```
  </Tab>
  
  <Tab title="Electron + Renderer">
    ```bash
    npm -C electron run dev
    ```
    
    Note: Electron main proxies all backend HTTP calls and attaches the auth token.
  </Tab>
</Tabs>

## Environment variables

Key environment variables from `.env.example`:

| Variable | Required | Description |
|----------|----------|-------------|
| `GOOGLE_API_KEY` | Yes | Gemini API key from AI Studio |
| `STELLARIS_API_TOKEN` | Auto | Backend auth token (auto-generated by dev.sh) |
| `STELLARIS_DB_PATH` | No | SQLite history database path (default: `./stellaris_history.db`) |
| `STELLARIS_SAVE_PATH` | No | Override auto-detected save path |
| `PARSER_BINARY` | No | Custom Rust parser binary location |
| `STELLARIS_LOG_LEVEL` | No | Logging level: DEBUG, INFO, WARNING, ERROR |

## Testing

### Python tests

Run the full test suite:

```bash
pytest tests/
```

Run specific test files:

```bash
pytest tests/test_signals.py -v
```

Run with coverage:

```bash
pytest tests/ --cov=backend --cov=stellaris_save_extractor
```

### Test markers

From `pyproject.toml:131`:

- `@pytest.mark.slow` - Marks tests as slow (skip with `-m "not slow"`)
- `@pytest.mark.integration` - Integration tests requiring real save files

### Python compilation check

Quick syntax validation without running tests:

```bash
python3 -m compileall -q backend stellaris_companion stellaris_save_extractor
```

### Renderer build test

Verify the React renderer builds successfully:

```bash
npm -C electron/renderer run build
```

### Main process syntax check

```bash
node --check electron/main.js
```

## Code quality

### Python linting and formatting

The project uses [Ruff](https://docs.astral.sh/ruff/) for linting and formatting:

```bash
# Check for issues
ruff check .

# Check formatting
ruff format --check .

# Auto-fix issues
ruff check --fix .

# Auto-format code
ruff format .
```

### Ruff configuration

From `pyproject.toml:67`:

<AccordionGroup>
  <Accordion title="Enabled rules">
    - `E` - pycodestyle errors
    - `W` - pycodestyle warnings

</AccordionGroup>

## Building for distribution

### PyInstaller backend bundle

Create a standalone backend executable:

```bash
pyinstaller --clean stellaris-backend.spec
```

The bundled executable will be at `dist/stellaris-backend`.

### Electron application

Build the complete Electron app with packaged backend:

```bash
npm -C electron run build
npm -C electron run dist
```

This creates platform-specific installers in `electron/dist/`.

## Common development tasks

### Viewing backend logs

Backend logs go to stdout by default. Set `STELLARIS_LOG_DIR` for file logging:

```bash
export STELLARIS_LOG_DIR=./logs
python3 -m backend.electron_main
```

### Testing with a specific save file

```bash
export STELLARIS_SAVE_PATH=/path/to/your/save.sav
./dev.sh
```

### Debugging parser issues

Test the Rust parser directly:

```bash
./stellaris-parser/target/release/stellaris-parser extract-save \
  /path/to/save.sav \
  --sections meta,player \
  --output - | jq .
```

### Cleaning build artifacts

```bash
# Python
rm -rf build/ dist/ *.egg-info __pycache__/

# Rust
cd stellaris-parser && cargo clean && cd ..

# Node
rm -rf electron/node_modules electron/renderer/node_modules
```

## Contribution workflow

<Warning>
Before contributing, review the current priorities in `CONTRIBUTING.md`. The project is currently focusing on DLC coverage, cross-platform reliability, and stability fixes.
</Warning>

### Commit message format

Use conventional commit format from `CONTRIBUTING.md:55`:

- `feat:` - New feature
- `fix:` - Bug fix
- `docs:` - Documentation only
- `refactor:` - Code change that neither fixes a bug nor adds a feature
- `test:` - Adding or updating tests
- `chore:` - Maintenance tasks

Examples:

```bash
git commit -m "fix: handle missing leader traits in extraction"
git commit -m "feat: add support for Astral Planes DLC"
git commit -m "docs: clarify session mode usage in rust-parser guide"
```

### Pull request process

<Steps>
  <Step title="Create branch">
    Create a branch from `main` with a descriptive name
  </Step>
  
  <Step title="Make changes">
    Implement your changes following the code style guidelines
  </Step>
  
  <Step title="Run tests">
    Ensure all tests pass and the Rust parser builds:
    ```bash
    pytest tests/
    cd stellaris-parser && cargo build --release && cd ..
    ruff check .
    ```
  </Step>
  
  <Step title="Verify alignment">
    Confirm the change aligns with current priorities (DLC/platform/stability)
  </Step>
  
  <Step title="Submit PR">
    Submit a pull request with a clear description of the changes
  </Step>
</Steps>

## Debugging tips

### Backend debugging

Add breakpoints using the Python debugger:

```python
import pdb; pdb.set_trace()
```

Or use VS Code's debugger with a launch configuration:

```json .vscode/launch.json
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Python: Backend",
      "type": "python",
      "request": "launch",
      "module": "backend.electron_main",
      "justMyCode": false
    }
  ]
}
```

### Electron debugging

Open DevTools in the Electron window:

- macOS: `Cmd + Option + I`
- Windows/Linux: `Ctrl + Shift + I`

For main process debugging, use `--inspect` flag:

```bash
electron --inspect=5858 electron/main.js
```

### Rust parser debugging

Enable debug logging:

```bash
RUST_LOG=debug ./stellaris-parser serve --path save.sav
```
