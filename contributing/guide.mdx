## What we're prioritizing right now

To keep the app stable at launch, we are currently accepting contributions that focus on:

- DLC coverage and correctness
- Windows/Linux reliability fixes
- Stability and polish for existing features (bugs, regressions, tests, docs)

Please avoid net-new feature work for now unless it is required to fix a critical bug or platform issue. If you're unsure whether a change fits, open an issue first and we will help scope it.

## Development setup

<Steps>
  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/gitmaan/stellaris-companion.git
    cd stellaris-companion
    ```
  </Step>

  <Step title="Create and activate virtual environment">
    ```bash
    python -m venv venv
    source venv/bin/activate  # On Windows: venv\Scripts\activate
    ```
  </Step>

  <Step title="Install dependencies">
    ```bash
    pip install -e .
    ```
  </Step>

  <Step title="Build the Rust parser">
    The Rust parser is required for the application to function.
    
    ```bash
    cd stellaris-parser
    cargo build --release
    cd ..
    ```
  </Step>

  <Step title="Configure environment variables">
    ```bash
    cp .env.example .env
    # Edit .env with your GOOGLE_API_KEY
    ```
  </Step>
</Steps>

<Note>
  You'll need a Gemini API key from [Google AI Studio](https://aistudio.google.com/) to use the application.
</Note>

## Running tests

Run the test suite to verify your changes:

```bash
# Run all tests
pytest tests/

# Run specific test file
pytest tests/test_signals.py -v
```

## Code style

Follow these conventions when contributing code:

- Format Python code with `ruff format`
- Use type hints for function signatures
- Follow existing patterns in the codebase

<Tip>
  Run `ruff check .` and `ruff format --check .` before submitting your pull request to ensure code quality.
</Tip>

## Commit messages

Use conventional commit format for all commits:

| Prefix | Description |
|--------|-------------|
| `feat:` | New feature |
| `fix:` | Bug fix |
| `docs:` | Documentation only |
| `refactor:` | Code change that neither fixes a bug nor adds a feature |
| `test:` | Adding or updating tests |
| `chore:` | Maintenance tasks |

<CodeGroup>
```bash Good examples
feat: add support for Machine Intelligence ethics
fix: resolve crash on Windows when parsing large saves
docs: update installation requirements for Rust toolchain
```

```bash Bad examples
updated some files
fixes
WIP
```
</CodeGroup>

## Pull request process

<Steps>
  <Step title="Create a branch from main">
    Start with a clean branch from the latest `main`:
    
    ```bash
    git checkout main
    git pull origin main
    git checkout -b your-feature-branch
    ```
  </Step>

  <Step title="Make your changes">
    Implement your changes following the code style guidelines and ensuring all tests pass.
  </Step>

  <Step title="Verify your work">
    Before submitting, ensure:
    - Tests pass: `pytest tests/`
    - Rust parser builds: `cd stellaris-parser && cargo build --release`
    - Code is formatted: `ruff format .`
  </Step>

  <Step title="Confirm alignment with priorities">
    Verify that your change aligns with current priorities:
    - DLC coverage and correctness
    - Platform reliability (Windows/Linux)
    - Stability and polish for existing features
  </Step>

  <Step title="Submit your pull request">
    Create a PR with a clear description that:
    - Explains what problem it solves
    - Describes the approach taken
    - Notes any breaking changes or migration steps
    - References related issues
  </Step>
</Steps>

<Warning>
  Net-new feature work is currently de-prioritized unless it unblocks a critical bug or platform reliability issue.
</Warning>
